# API response types mirror the wire format (snake_case); domain types stay camelCase; mapping happens at the boundary

The Frame CLI consumes a snake_case JSON API. TypeScript interfaces describing those responses (`MeResponse`, and every codegen'd resource response that follows) declare their fields exactly as the wire format does — `merchant_id`, `dev_mode`, `api_version` — without case translation. The CLI's *internal* types (`Credential`, banner contexts, fixture domain models, anything not directly representing an HTTP response body) keep camelCase, matching the rest of the TypeScript codebase. Translation between the two conventions is an explicit, visible mapping that happens once per response, at the call site that consumes it (`src/commands/login.ts`, `src/commands/whoami.ts`, and any future codegen'd command). There is no automatic snake↔camel transform anywhere in the request pipeline.

## Considered options

- **CamelCase everywhere, with a recursive `camelize` helper applied to every API response in `auth/api-client`** — rejected. The helper is a hidden dependency: a bug or omission silently corrupts every endpoint, and the failure mode is `undefined` at runtime with no compile error. We hit exactly this failure mode by accident during the `MeResponse` shape-drift incident (the original `{ id: string; name: string }` type didn't match the wire's `merchant_id` / `merchant_name`, and the resulting "Logged in as undefined (undefined)" message was indistinguishable from a missing field on the server). With a hidden mapper, that class of bug recurs every time a field is added to the API and someone forgets to thread it through.
- **CamelCase everywhere, with hand-written per-endpoint mappers (`parseMe`, `parseTransfer`, …) in `auth/api-client`** — rejected. Less risky than the recursive helper, but it just relocates the boilerplate from the call site (which knows what fields it cares about) to a parser layer that doesn't (and therefore tends to map fields nobody uses, plus drift independently of the consumer). The CLI is a thin shell over the API; pushing the mapping further from the consumer makes drift harder to spot in review.
- **Generate camelCase types from the OpenAPI spec via a `casingTransform` step in the codegen pipeline** — rejected. The vendored OpenAPI spec at `vendor/openapi/openapi.yaml` is the source of truth for the v1 codegen'd resource subcommands (per `docs/plans/cli-v1-client.md`). Codegen's natural output is snake_case. Adding a transform step means the generator's output diff against the spec is no longer mechanical — every spec change requires re-deriving the casing table. The transform itself is also a piece of code that can drift (the `merchant_id_2fa` → `merchantId2fa` vs `merchantId2Fa` debate happens once per project that does this).
- **Snake_case at the API boundary, camelCase internally, mapping at the call site** — chosen. Response types diff mechanically against API docs and the OpenAPI spec; the codegen pipeline emits straight from the spec without a transform; the mapping line `merchant: me.merchant_id` is right next to the consumer that needs it, so a missing field is a compile error rather than a runtime `undefined`; the boundary is explicit and greppable (`grep -r 'me\\.\\(merchant_id\\|merchant_name\\|dev_mode\\)'` lists every call site in seconds).

## Consequences

- Response interfaces in `src/auth/api-client.ts` (and any future `src/commands/resources/*` codegen'd response types) use snake_case field names. Internal types in `src/auth/keyring.ts`, `src/fmt/banner.ts`, and command-local types use camelCase. The convention is mechanical; reviewers should treat a mixed-case response interface (e.g. `merchantId` on a `*Response`) as a regression.
- ESLint's default config has no naming-convention rule active in this repo, so the snake_case fields on response types compile and lint cleanly today. If a future PR enables `@typescript-eslint/naming-convention`, the rule must exempt `Property` declarations on identifiers matching `*Response` (or, more loosely, the `src/auth/api-client.ts` and `src/commands/resources/**/*.ts` paths).
- Mapping is explicit at every boundary. Adding a new field to a response type without consuming it is fine — the field is just declared. Consuming a field requires writing a line that names it on both sides (`merchant: me.merchant_id`), which is the forcing function: a missing field is a TypeScript error, not a runtime `undefined`. A field that disappears from the wire surfaces as a 404-style 422 from the server, not a silent `undefined`.
- The codegen pipeline (when it lands) emits response types verbatim from the OpenAPI spec. Drift between the spec and the generated types is a one-line diff, not a transform-table debugging exercise.
- Tests that fixture API responses (`vi.fn().mockResolvedValueOnce({ merchant_id: ..., dev_mode: true, ... })`) use snake_case for the mocked response and camelCase for any mocked credential or banner context. Reviewers should expect both conventions in a single test file and not "fix" them to one or the other.
- This decision is hard to reverse: changing it later means rewriting every response type, every test fixture, and every mapping line, plus deciding what to do with the codegen pipeline's output. We are committing to it now, while the surface is small.
- Stripe's official Node SDK (`stripe-node`) follows the same convention for the same reasons; this is the well-trodden path for thin TypeScript clients over snake_case JSON APIs.
